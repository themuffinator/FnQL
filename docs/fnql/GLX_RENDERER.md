# GLx Renderer Architecture Note

## Status

GLx is FnQL's canonical OpenGL-lineage renderer module. Normal modular builds include it through `USE_GLX`; `opengl` remains the release default until the GLx promotion gate is green.

The initial release-candidate gate policy is defined in [GLX_RC_GATES.md](GLX_RC_GATES.md). GLx can be selected explicitly for validation and diagnostics today; default renderer promotion is separately guarded by [GLX_PROMOTION.md](GLX_PROMOTION.md) and `scripts/glx_promotion.py`.

The final replacement contract is defined in [GLX_FINAL_CONTRACT.md](GLX_FINAL_CONTRACT.md). This page describes the current transitional implementation; where compatibility fallback language conflicts with that ADR, the ADR governs future GLx replacement work.

The current feature-closure ledger is [GLX_FEATURE_MATRIX.md](GLX_FEATURE_MATRIX.md). It records whether each legacy OpenGL and FnQL feature surface is `covered`, `partially covered`, or `missing`, along with the evidence and closure gate for each row.

The current source-coupling ledger is [GLX_LEGACY_COUPLING.md](GLX_LEGACY_COUPLING.md). It records every legacy `code/renderer/*.c` translation unit still compiled into GLx, the compatibility role that keeps it there, and the extraction target that should shrink it. The promotion script checks the ledger against Meson, Make, and MSVC build inputs so P2 source-coupling reduction can be ratcheted instead of guessed.

## Decision

GLx keeps FnQL's existing renderer ABI intact:

- `REF_API_VERSION` is 13 after the versioned WebUI, QL host-font, and
  renderer-only liquid-interaction exports; every in-tree renderer implements
  the same boundary.
- `GetRefAPI` remains the only exported renderer entry point.
- The engine continues to communicate through `refimport_t` and `refexport_t`.
- `cl_renderer glx` loads the external renderer module named `fnql_glx_<arch>` from the single FnQL client executable.

The first implementation slice deliberately reuses the compatibility-proven OpenGL renderer as the rendering baseline and compiles it with `RENDERER_GLX`. GLx-specific code lives in `code/rendererglx/` and starts with capability detection, debug callback wiring, timer-query profiling, static-world cache telemetry, dynamic-stream strategy selection, and build-system integration.

## Product Tiers

GLx records one active product tier at OpenGL initialization. The tier is always one of the five final replacement targets:

- `GL12`: fixed-function compatibility floor for stock gameplay and demo-safe rendering.
- `GL2X`: baseline programmable tier for OpenGL 2.x / GLSL-era systems.
- `GL3X`: OpenGL 3.x modern-performance tier.
- `GL41`: macOS-ceiling modern tier without dependence on 4.3+ conveniences.
- `GL46`: high-end Windows/Linux tier for the full 4.6-class feature surface.

Feature flags remain recorded underneath the product tier, and GLx also keeps an internal capability hint (`fixed-function`, `programmable`, `modern`, or `high-end`) for accelerator choices. Hot renderer paths should consume the one-time product tier plus the frozen feature table instead of scattering extension checks through draw submission.

The `GL12` tier now has an explicit fixed-function executor policy inside GLx. It accepts the deterministic pass schedule and the same render-product vocabulary as the higher tiers, but its supported runtime surface is deliberately conservative: client-memory draw submission, fixed-function material state, lightmaps, multitexture/lightmap composition when available, fog, sprites, beams, basic dynamic lights, stencil shadows when the active context provides stencil support, screenshots, and demo-safe SDR output. It does not require or advertise the GLSL material compiler, dynamic stream VBO uploads, scene-linear HDR, tone mapping, color grading, bloom post-chain execution, or indirect/multidraw static-world submission.

The `GL2X` tier is the first programmable executor policy. It keeps client-memory draw submission as a compatibility fallback, but its intended path is GLSL-era material execution over transient stream uploads for common prepared id Tech 3 stage shapes, dynamic entities, lightmaps, multitexture, fog, sprites, beams, screenshots, demos, and a postprocess-lite surface such as gamma/resolve and bloom parity. It explicitly does not require UBO-backed frame state, persistent mapped uploads, sync-object-only allocation, scene-linear HDR output, tone mapping, color grading, DSA, indirect draws, or multidraw-indirect static-world submission.

The `GL3X` tier is the first performance executor policy. It requires the modern-feeling GLx path: FBO-backed postprocess, UBO-style frame/object constants, timer-query profiling, sync-aware map-range uploads, GLx-owned static and dynamic buffer products, common GLSL material execution, dynamic entities, the modern post chain, scene-linear output, screenshots, and demos. It still rejects persistent mapped buffer-storage uploads, direct-state-access requirements, and indirect/multidraw submission requirements so OpenGL 3.x remains a shipped tier rather than a disguised GL4 path.

The `GL41` tier is the named macOS-ceiling executor policy. It is a fully supported modern tier, not a degraded high-end path: it keeps FBO postprocess, UBO-style constants, timer queries, sync-aware uploads, static/dynamic buffer ownership, modern post-chain, scene-linear output, high-quality SDR, optional platform HDR output, screenshots, and demos. It explicitly does not require GL4.3 debug output, GL4.4 buffer storage or persistent mapping, GL4.5 direct state access, or multi-draw-indirect submission.

The `GL46` tier is the named high-end executor policy for Windows/Linux-class 4.6 drivers. It requires persistent mapped uploads through buffer storage, sync-heavy streaming, direct state access, multi-draw-indirect submission, aggressive static-world submission, detailed GPU counters, scene-linear post, hardware HDR output, screenshots, and demos. Its diagnostics include both render-product counters and static-world MDI call/index counters so high-end gains remain measurable instead of assumed.

## Current Foundation

The bootstrap modules currently provide:

- `renderer_switch <renderer> [fast|keep_window|full|destroy_window]` in dynamic-renderer builds for runtime renderer reload/switch testing through the same lifecycle as `vid_restart`.
- `glxinfo` / `glxcaps` command output.
- `glxmaterial` command output for GLSL material program readiness, compile/link/precache failures, cache hits, bind counts, and the last material request.
- `glxpostprocess` command output for FBO readiness, internal render target sizing, scene-linear HDR state, internal precision, render-scale and greyscale controls, postprocess frame-feature counters, output transform state, final gamma/tone-scale output, bloom chain creation, bloom pass mode, and screen-map/MSAA/SSAA blits.
- `glxstaticworld [limit]`, `glxstaticworld hot [limit]`, `glxstaticworld commands [limit]`, and `glxstaticworld spans [limit]` command output for static-world cache, arena, queue, packet item ranges, draw-range inspection, packet heat ranking, indirect command inspection, and MDI span diagnostics.
- `glxstreamtest` developer command for exercising one dynamic stream reservation/upload/commit without changing scene rendering.
- `r_glxDebug` and `r_glxDebugVerbose` cvars for platform debug-context requests plus debug-output callback setup when the driver exposes `KHR_debug` or `ARB_debug_output`.
- `r_glxDebugGroups` for optional `KHR_debug` groups around GLx-observed shader batch submission.
- `r_glxGpuTiming` for non-blocking backend GPU timing with timer queries when supported.
- `r_glxGpuPassTiming` for non-blocking pass-level GPU timings on postprocess, bloom, blits, copies, and flare rendering when timestamp queries are available; `r_speeds 7` also reports blit, bind, clear, and full-screen pass counts.
- `r_glxPostProcessDebug` for optional logging of GLx-observed FBO, render-scale, gamma, and bloom parity events.
- `r_glxWorldRenderer` for enabling the compatibility-first GLx static world renderer bundle with per-draw fallback.
- `r_glxStaticWorldArena` for GLx-owned uploads of the packed legacy static world VBO/IBO payload.
- `r_glxStaticWorldArenaDraw` for binding the GLx-owned static arena during legacy VBO draws.
- `r_glxStaticWorldDraw` for GLx submission of legacy-prepared static device-index runs.
- `r_glxStaticWorldSoftDraw` for routing static-world CPU-index fallback runs through the GLx dispatcher.
- `r_glxStaticWorldDrawPolicy` for restricting GLx static draws to `full`, `contained`, or `all` packet ranges.
- `r_glxStaticWorldMultiDraw` for batching same-state static device-index runs with `glMultiDrawElements`.
- `r_glxStaticWorldPacketBatch` for ordered full-packet static batch spans that submit from manifest-owned packet offsets before falling back to legacy run spans.
- `r_glxStaticWorldIndirectBuffer` for capability-gated upload of the static packet command manifest into a GL-owned indirect command buffer.
- `r_glxStaticWorldIndirectDraw` for capability-gated submission of full manifest-packet static draws through `glDrawElementsIndirect` when the command buffer is ready.
- `r_glxStaticWorldMultiDrawIndirect` for capability-gated submission of contiguous full-packet static command spans through `glMultiDrawElementsIndirect`.
- `r_glxStaticWorldMultiDrawIndirectCompact` for developer-only upload of compact visible full-packet command batches when static command-buffer slots are not adjacent or not uploaded.
- `r_glxStaticWorldMultiDrawIndirectSpans` for splitting filtered static batches into ordered MDI and fallback multidraw spans.
- `r_glxStreamMode` and `r_glxStreamMegabytes` for selecting and allocating the transient geometry stream ring.
- `r_glxStreamTess` for developer-only shadow uploads of legacy tessellation payloads through the GLx stream ring.
- `r_glxStreamDraw` for the profile-controlled streamed draw path for eligible generic shader stages.
- `r_glxStreamDrawKeyMode` for narrowing or widening the streamed draw material-key allowlist during experiments.
- `r_glxStreamDrawMultitexture` for separately gating eligible fixed-function multitexture stages into streamed draws.
- `r_glxStreamDrawFog` for separately gating fog-only passes into streamed draws.
- `r_glxStreamDrawDepthFragment` for separately gating eligible depth-fragment stages into streamed draws.
- `r_glxStreamDrawTexMods` for separately gating CPU-computed texmod stages into streamed draws.
- `r_glxStreamDrawEnvironment` for separately gating CPU-computed environment texture-coordinate stages into streamed draws.
- `r_glxStreamDrawDynamicLights` (`0`, `1`, or `auto`) for the controlled dynamic-light stream path, `r_glxDlightScissor` (`0`, `1`, or `auto`) for projected-dlight overlay scissor rectangles, plus `r_glxStreamDrawScreenMaps` and `r_glxStreamDrawVideoMaps` for explicit high-risk material-key experiments that stay off in the RC profile.
- `r_glxStreamDrawShadows` for profile-controlled stencil shadow-volume streamed draw submission.
- `r_glxStreamDrawBeams` for profile-controlled immediate beam entity draw-array streamed submission.
- `r_glxStreamDrawPostProcess` for profile-controlled fullscreen postprocess draw-array streamed submission.
- `r_glxRequireOwnership` for proof runs that reject GLx legacy-delegation draw submissions instead of allowing compatibility-prepared paths to render successfully.
- `r_glxMaterialRenderer` for using the independent GLSL material program path on streamed GLx draws when the driver can compile the complete compatibility shader set.
- `r_glxMaterialPrecache` for compiling the built-in GLx material permutation coverage at OpenGL startup so common shader-language failures fall back before the first draw.
- `r_glxMaterialDebug` for material shader diagnostics, with level `2` dumping failed GLSL source.
- `r_glxPostShaderCache`, `r_glxPostShaderPrecache`, `r_glxPostShaderDebug`, and `r_glxPostShaderExecute` for compiling, measuring, and experimentally binding generated GLx post/output shader programs on eligible display-referred SDR legacy-gamma, scene-linear final, bloom, blit/capture, greyscale, and hardware-HDR paths. Execution is off by default and falls back to the legacy ARB path when the frame is not eligible.
- `r_speeds 7` output for GLx capability, locked pass-schedule text/hash, backend timer, shader-batch counts, draw counts, postprocess/FBO/bloom counters, stream readiness/upload counters, per-category dynamic stream metrics, static-world cache counters, packet lookup counters, and indirect-buffer readiness.
- GLx ownership diagnostics in `glxinfo` and `r_speeds 7` that count legacy draw delegation by generic indexed draws, VBO device draws, VBO soft draws, direct draw-array fallbacks, and modern-tier post/output planning. The post/output line reports planned node/output counts, executable node/output counts, stable fingerprints for the last GLx-consumed `PostNode`, `OutputTransform`, and planned post/output graph, plus an explicit fallback mask. These counters are the Task B cutover bar: final GLx promotion requires legacy draw delegation to stay at zero and GL3X+/GL41/GL46 post/output products to report executable GLx-owned counts with zero fallback.
- Static-world VBO cache, packet-manifest, and queue-split accounting, shader-batch telemetry, and material-stage key telemetry from the existing OpenGL renderer paths.

This preserves compatibility while creating the measurement and capability surface needed for later GLx-only work.

## Implementation Map

The current C++ boundary is intentionally small and split by ownership:

- `code/renderercommon/tr_glx_public.h`: shared GLx compatibility vocabulary for draw-path ids, batch/stage flags, material combine ids, postprocess results, stream skip reasons, and public stream reservations. This keeps pure GLx classification logic and renderer-side compatibility hooks from depending on the full module ABI just to agree on counters and payload shapes.
- `code/renderercommon/tr_glx_api.h`: renderer-common declarations for the GLx C bridge implemented by `code/rendererglx/`. It forward-declares the legacy ABI pointer types it needs instead of including `tr_public.h`, so `code/renderer` can call the bridge without including GLx implementation headers.
- `code/renderercommon/tr_glx_bridge.h`: renderer-common forwarding facade for the GLx C bridge. It centralizes lifecycle handoff, frame timing, draw submission, stream reservation/upload calls, and material/FBO/postprocess/static-world telemetry without including `tr_local.h`, `tr_public.h`, `qgl`, or GL object typedefs; non-GLX builds get explicit no-op stubs rather than implicit renderer substitution.
- `code/renderer/tr_glx_compat.h`: renderer-owned compatibility adapter between the legacy OpenGL substrate and the GLx module. Legacy OpenGL files include this local boundary instead of reaching directly into `code/rendererglx/`; it now contains the legacy-only pieces that still need `tr_local.h`, fixed-function state setup, or shader-stage conversion to the shared GLx material-key vocabulary. In GLx builds, final `glDrawElements` / `glDrawArrays` submission goes through the GLx bridge instead of direct `qglDraw*` calls in this adapter.
- `glx_caps.*`: one-time OpenGL version, extension, debug-context state, feature, product-tier, and internal capability-hint detection.
- `glx_types.h`, `glx_caps_logic.h`, `glx_material_key.h`, `glx_render_ir.h`, and `glx_stream_logic.h`: renderer-independent GLx enums, render-product contracts, and pure classification/fallback logic used by both runtime code and tests without pulling in OpenGL function declarations or module imports. Legacy fixed-function texture environment enums are translated to GLx material combine ids at `tr_glx_compat.h` before the pure material-key classifier sees them.
- `glx_debug.*`: debug-output callback wiring, `KHR_debug` object labels/groups, and notification filtering.
- `glx_draw.*`: GLx-owned low-level draw-submission edge for indexed and array draws. It resolves the core draw entry points inside the GLx module and is called by the active executor rather than directly by renderer-common code.
- `glx_executor.*`: render-product executor surface for the five final GLx tiers (`GL12`, `GL2X`, `GL3X`, `GL41`, and `GL46`). Each tier consumes the same `FramePass`, `WorldPacket`, `DynamicDraw`, `MaterialIR`, `UploadPlan`, `PostNode`, and `OutputTransform` products; the GLx front end emits the canonical `FramePass` schedule once at OpenGL startup, including a native `dynamic-lights` slot after opaque world/entity work and before transient dynamic-scene effects, the executor validates and records it, and the current runtime routes final dynamic draw submission through `DynamicDraw` IR while recording consumption/reject counters for the other products as their front-end telemetry is emitted. The `GL12` executor is a named fixed-function policy: it submits client-memory indexed/array draws through GLx-owned `glx_draw.*`, disables stream VBO uploads at stream initialization, treats GLSL materials and modern post/HDR nodes as unavailable, and reports the supported fixed-function surface through `glxinfo` plus `r_speeds 7`. The `GL2X` executor is a named programmable policy: it accepts transient stream upload products, GLSL material products, and postprocess-lite nodes, rejects persistent-mapped/sync-required uploads and scene-linear/modern post products, and reports its programmable support and unavailable modern features in the same diagnostics. The `GL3X` executor is a named performance policy: it requires FBO postprocess, UBO-style frame/object constants, timer queries, sync-aware uploads, static/dynamic buffer ownership, modern post-chain, scene-linear output, screenshots, and demos, while rejecting GL4-only persistent mapped uploads, direct-state-access requirements, and indirect submission requirements. The `GL41` executor is a named mac-modern policy: it provides the supported macOS ceiling path while reporting GL4.3 debug output, GL4.4 buffer storage/persistent uploads, GL4.5 DSA, and MDI as unavailable requirements rather than architectural dependencies. The `GL46` executor is a named high-end policy: it requires persistent uploads, buffer storage, sync-heavy streaming, DSA, MDI, aggressive static-world submission, detailed GPU counters, and hardware HDR output, and reports compact high-end counters for runtime proof.
- `glx_profiler.*`: backend frame counters, rotating timer-query collection, shader-batch/material telemetry, and draw-call/index counters.
- `glx_postprocess.*`: compatibility FBO lifecycle telemetry, render-scale/resolve counters, final gamma result tracking, and bloom parity accounting for the shared OpenGL postprocess path.
- `glx_post_shader.*`: generated GLx post/output GLSL cache and opt-in final-output binder. It compiles shader programs from the `glx_post_shader_source.h` feature key, tracks source hashes, cache hits/misses, compile/link/source failures, representative startup precache, and final-pass eligibility/bind/reject counters. `r_glxPostShaderExecute 1` can bind the cached GLSL program for display-referred SDR legacy gamma, scene-linear SDR, bloom-final, bloom-prefinal, greyscale, blit/capture, Display P3/EDR, scRGB/linear HDR, and HDR10/PQ output, including lift/gamma/gain, Bradford white-point, LUT atlas, bloom, legacy gamma/overbright, and output-headroom uniforms. Disabled or rejected shapes continue through the legacy ARB executor.
- `glx_static_world.*`: static BSP/VBO cache accounting, GLx-owned packet manifest snapshots, optional static arena uploads/draw binding, optional GLx device-run submission, and prepared-queue split telemetry.
- `glx_stream.*`: dynamic stream policy selection, buffer lifecycle, reservation/upload/commit API, per-category dynamic-scene counters, and allocator counters for persistent-map, map-range, or orphan/subdata paths.
- `glx_module.*`: the implementation side of the C ABI bridge used behind the renderer compatibility facade. New legacy-renderer touch points should be declared in `renderercommon/tr_glx_api.h`, forwarded through `renderercommon/tr_glx_bridge.h` when they are renderer-neutral, and kept in `tr_glx_compat.h` only when they need legacy renderer state. Shared ids and small public payload structs belong in `renderercommon/tr_glx_public.h`.

`r_glxStreamMode auto` selects `persistent-map` only when buffer storage and sync objects are present, falls back to `map-range` when available, and otherwise uses the portable orphan/subdata path. Forced modes fall back rather than failing renderer initialization. The stream ring is allocated during OpenGL initialization and now exposes reserve/upload-at/commit primitives so one draw can safely upload several attribute ranges inside one contiguous reservation. When sync objects are available, frames that touched the stream insert a GL fence and the next stream reservation waits before reusing the reset frame range. Same-frame stream wraps are rejected instead of reusing the beginning of the ring while earlier draws from that frame may still be in flight; the caller returns to the GLx draw-submission bridge with legacy-prepared client arrays, and the reject is counted separately from ordinary wraps. Non-streamed scene state prep still uses the legacy client-array path by default, but GLx builds own the final draw-submission edge.

Streamed draw submissions now preserve `GL_ARRAY_BUFFER` and `GL_ELEMENT_ARRAY_BUFFER` through a GLx-owned binding cache exposed by the renderer-common bridge. The compatibility adapter no longer issues a `GetIntegerv` around each eligible streamed draw; GLx queries only when the cache is unknown, uses the same tracked handoff for stream reservation/upload/map-unmap/delete internals, mirrors direct legacy VBO buffer binds into the cache through the bridge, invalidates the cache at frame completion, and reports query/hit/restore counters plus the tracked array/element bindings in `glxstream` and `r_speeds 7`.

On `GL12`, dynamic streaming is intentionally not initialized. The stream diagnostic reason becomes `GL12 fixed-function tier uses client-memory draw submission`, and any accidental transient-stream render product is rejected by the GL12 executor policy instead of being treated as supported. That keeps stock gameplay and demos on the fixed-function path without requiring VBO-era buffer objects.

`r_glxStreamTess 1` enables shadow uploads from `RB_EndSurface`: after the legacy stage iterator has drawn a non-VBO tessellation batch, GLx uploads the batch's vertex and index payloads through the selected stream strategy and records upload pressure. The uploaded data is not used for drawing yet, so this mode is for development telemetry rather than visual parity testing.

`r_glxStreamDraw 1` enables the guarded streamed draw path used by the RC profile. With only the parent cvar enabled, the path still starts from generic, single-texture, non-depth-fragment shader stages; the RC profile then opts in the proven material-stage gates and state-only draw-array submissions independently. The legacy renderer still computes colors, texcoords, state bits, texture binding, fog, dynamic lights, and fallback behavior; GLx uploads the already-computed xyz/color/texcoord arrays and index list into one stream reservation, binds that buffer for vertex and element arrays, draws from stream offsets through the GLx draw API, then restores the legacy CPU vertex/color/texcoord pointers. If any stream reservation or upload fails, the stage returns to `R_DrawElements`, which is now a GLx-owned legacy-delegation draw in GLx builds rather than a direct compatibility-renderer draw.

Every streamed dynamic batch carries one or more dynamic-scene category bits through `tr_glx_public.h`, the renderer-common bridge, and `glx_stream.*`. Generic tessellation draws infer entity, first-person weapon, UI, particle, poly, mark, dlight, or special-pass identity from the active entity, projection state, surface type, shader sort, polygon-offset state, and material flags. Dlight-map work is now tagged as its own overlapping category instead of being folded into `special`, so an entity lit by a streamed dlight can count as both `entity` and `dlight`. The same material flags and category masks now also classify each streamed draw into a small GLx `DynamicDrawRole` (`generic`, `dlight`, `shadow`, `beam`, or `postprocess`) for IR-facing diagnostics, without changing submission behavior; `dlight` role products default to the native `dynamic-lights` frame-pass slot for later shaderized accumulation work. Stream paths use classified GLx draw wrappers so the final `DynamicDraw` product receives the same role/pass identity as the stream-reservation telemetry. The executor records accepted dynamic draw products by role and by frame-pass placement as draw/index/vertex triples, giving native dlight products a separate proof surface from stream-reservation telemetry. State-only helpers such as immediate beams, stencil-shadow quads, and postprocess quads pass explicit category masks into the same stream reservation/upload/commit helpers. `glxinfo` reports category and role draw/attempt/fallback totals plus executor role/pass product totals, and `r_speeds 7` emits compact category, role, and executor-role samples for performance baselines.

`r_glxStreamDrawMultitexture 1` additionally allows eligible fixed-function multitexture stages through the same stream path. In that mode GLx uploads both texture coordinate bundles, binds both client texture-coordinate arrays from the stream buffer, and then restores the legacy unit-0 and unit-1 pointers. This is separately gated because it touches more GL client state and should be parity-tested independently from the single-texture path.

`r_glxStreamDrawFog 1` additionally allows the classic fog overlay pass to draw from the stream ring. The legacy renderer still computes fog colors, fog texture coordinates, texture binding, blend state, and depth-function selection; GLx only uploads the prepared xyz/color/fog-texcoord arrays plus indexes, draws from stream offsets, and restores the previous array and element buffer bindings before returning. This is separately gated because fog is a compatibility-visible overlay and should be checked independently from normal material stages.

`r_glxStreamDrawDepthFragment 1` additionally allows eligible `depthFragment` shader stages through the stream path, including multitexture stages when `r_glxStreamDrawMultitexture 1` is also enabled. The legacy shader collapse path no longer rejects a `depthFragment` base stage solely because it can be combined with a compatible second texture stage. GLx uploads the prepared stage arrays once, issues the normal draw from stream offsets, then mirrors the legacy second pass by forcing depth writes, enabling the existing ARB sprite fragment program, and drawing the same streamed indexes again before restoring buffer and client-array state. This is separately gated because it changes a two-draw compatibility path and should be parity-tested independently from plain material stages.

`r_glxStreamDrawTexMods 1` additionally allows shader stages whose texture coordinates were modified by the legacy CPU texmod pipeline to use the same stream path. GLx does not interpret texmod grammar or move that compatibility-sensitive math into GLSL here; it uploads the final texcoord arrays that `R_ComputeTexCoords` already produced. This is separately gated because texmods are common enough to matter but visible enough that they deserve their own parity switch and counters.

`r_glxStreamDrawEnvironment 1` does the same for stages using legacy CPU-computed environment texture-coordinate generation. The renderer still calculates the final coordinates from the existing tessellation normals before GLx sees the draw, so this is another explicit prepared-array allowlist rather than a shader grammar migration. `r_glxStreamDrawDynamicLights`, `r_glxStreamDrawScreenMaps`, and `r_glxStreamDrawVideoMaps` split the remaining broad material-key experiments into named gates and counters; the RC profile now uses the dynamic-light `auto` gate while screen-map and video-map streams stay disabled because those paths are more view- and content-sensitive.

`r_glxStreamDrawDynamicLights 1` now also covers the legacy dynamic-light overlay pass when `USE_LEGACY_DLIGHTS` is active and the ARB per-pixel lighting pass when `USE_PMLIGHT` is active. The lighting projection, triangle clipping, per-vertex colors or normals, texture coordinates, blend mode, ARB program setup, and light texture binding still come from the existing renderer; GLx only uploads the prepared dynamic-light xyz/color-or-normal/texcoord/index arrays into the stream ring and submits that pass from the stream buffer. `r_glxStreamDrawDynamicLights auto` is a controlled hardening gate that opens only when parent streamed draws are enabled, the stream ring is ready, and the detected GLx tier is GL3X, GL41, or GL46. The RC and stress profiles set this cvar to `auto`, while `1` remains an explicit developer override for lower-tier experiments. Runtime sweep diagnostics now require successful streamed dlight draws to appear in the render-IR dlight role and `dynamic-lights` pass counters, so the profile cannot silently route lighting back into generic dynamic-scene accounting.

Dynamic-light streamed draws now report their own attempt/draw/fallback counts, attempted and submitted upload bytes, index bytes, stream-ring wraps, same-frame wrap rejects, and fence wait/timeout/failure attribution in `glxstream`, `glxinfo`, and compact `r_speeds 7` samples. The native `r_glxDlightProgram` GLSL path also reports availability queries, cache hits, program creates/failures, binds, unbinds, and uniform updates, while dlight state telemetry records legacy overlay passes, dlight/fog/shadow texture binds, shadow-atlas FBO bind/restore counts, and state changes. Dlight build telemetry records legacy projected-light candidates, skipped lights, no-hit lights, touched vertices/indexes, emitted lit indexes, and promode fast-pass vertex/index pressure; the projected legacy path also reports per-surface light-bit culling for vertices/indexes avoided before texcoord/color generation. Projected legacy dlights additionally record conservative screen-space scissor candidates and coverage, with `r_glxDlightScissor auto` enabled by the RC/stress profiles to apply successfully computed rectangles during overlay draws while the standalone default remains `0`. Together these counters let dlight stress captures distinguish CPU build pressure, coarse-cull savings, stream-upload pressure, program-state churn, texture/FBO churn, and blend-coverage pressure before shaderized or packet-local dlight work begins.

`r_glxStreamDrawShadows 1` covers stencil shadow-volume passes and the final full-screen stencil darken quad. The compatibility renderer still projects the duplicated shadow-volume vertices, builds silhouette indexes, configures color writes, stencil state, culling, blend state, and texture state. GLx uploads the prepared xyz/index payload once, submits the same back-face increment and front-face decrement draws from the stream ring, uploads the final darken quad when shadows are applied, and falls back to the original locked client-array or draw-array paths if reservation or upload fails. This is part of the RC profile now and remains intentionally separate from material keys because shadow volumes are state-only geometry rather than shader-stage material draws.

`r_glxStreamDrawBeams 1` covers immediate beam entity geometry such as the simple red beam surface path. The compatibility renderer still derives the beam endpoints, perpendicular ring points, texture disable, additive blend state, color, and entity transforms; GLx only streams the prepared triangle-strip xyz array and submits the same draw-array primitive from the ring. This is part of the RC profile now and closes a narrow non-indexed dynamic-entity surface gap without folding draw-array state-only passes into the material permutation system.

`r_glxStreamDrawPostProcess 1` covers fullscreen postprocess quads routed through the ARB bloom/gamma `RenderQuad` helper and the colored bloom-reflection lens-effect triangle batch. The compatibility renderer still owns the FBO binding, program selection, viewport/projection setup, texture binding, texcoord layout, per-vertex reflection colors, and blend state; GLx only streams the prepared xyz/texcoord or xyz/texcoord/color arrays and submits the same draw-array primitives from the ring. This is part of the RC profile now and is kept separate from material keys because postprocess geometry is display pipeline work, not shader-stage scene material permutations.

`r_glxMaterialRenderer 1` makes streamed GLx draws bind GLSL material programs instead of relying on fixed-function texture combiners for the supported compatibility-stage shapes. GLx now compiles the prepared id Tech 3 stage language into `MaterialIR`, then validates that IR into a tier-visible material state plan that preserves the legacy sort value and carries the full GLSL program/state key. The shader cache is keyed by that full prepared language: combine mode (`single`, multitexture modulate/add/replace/decal, or fog), prepared-stage features, `rgbGen`, `alphaGen`, their wave-function opcodes, `tcGen`, ordered `tcMod` chains and operator masks for both bundles, `tcMod stretch` wave-function slots, detail-stage and other stage flags, raw state bits, decoded blend/depth/alpha-test state, fog color-adjust mode, dynamic-light/screen/video map flags, and fog status. The compact combine/features key still exists as the first part of that stage key, but it no longer collapses distinct stage-language shapes into one GLSL source program. Texture coordinate generation, texmods, colors, alpha test, texture binding, depth-fragment second-pass ARB program behavior, and fallback ordering still come from the legacy renderer before GLx uploads the already-computed arrays, keeping compatibility-sensitive math shared while GLx owns the broader GLSL permutation selection and diagnostics. Startup precache compiles the built-in RC combine/features set plus representative coverage for every public `rgbGen`, `alphaGen`, wave-function, `tcGen`, `tcMod`, ordered modifier-chain, fog-adjust, blend, depth, and alpha-test id so shader-language failures surface before normal play; observed runtime keys outside that coverage compile into their own full-stage permutations on demand. Unsupported stage-language combinations produce named compiler reasons in `glxmaterial`/`glxinfo` instead of disappearing into an anonymous material-key rejection. If generated GLSL for a stage permutation would exceed the fixed source buffer, GLx marks that material program unavailable and falls back instead of compiling truncated source.

The material compiler now also builds a GLx-native frame/object/material parameter block from every `MaterialIR` before selecting a translated stage key. The block records the sort/pass feature mask, object color/tcgen/wave inputs, and material state/texmod/fog/combine fields, and GLx plans shader state from that block rather than from the raw compatibility-shaped IR. The compatibility renderer still prepares texture coordinates, colors, and texture bindings, but the GLx material boundary is now a stable parameter block that future shader interfaces can consume directly. `glxmaterial` and compact performance samples report built and invalid block counts plus a stable parameter-block fingerprint; invalid blocks are gate failures, and modern-tier proof requires a non-zero block fingerprint alongside positive block counts.

Postprocess parity is implemented by keeping the compatibility-proven OpenGL FBO and ARB-program postprocess path as the fallback while GLx can opt in to generated shader ownership for the final/output edge. Under `RENDERER_GLX`, that shared path now reports FBO init/shutdown state, internal render/capture/window dimensions, HDR-capable FBO mode, scene-linear color mode, separate internal precision through `r_hdrPrecision`, internal render scaling, greyscale strength, MSAA resolve blits, supersample capture blits, screen-map copies, output transfer, tone-map operator, exposure, color-grading mode, minimized/screenshot fallback output, bloom chain allocation, bloom intermediate storage policy/format through `r_hdrBloomFormat`, and bloom execution. `r_hdr` now selects the high-precision FBO path; legacy tone mapping preserves Quake III display-referred lighting, while non-legacy tone mapping, color grading, and explicit HDR output opt into scene-linear color. `r_hdrPrecision` preserves the old debug/storage control, including automatic 8-bit SDR storage and explicit `RGBA16F` scene storage when `r_hdr 1`. `r_hdrBloomFormat` is deliberately narrower in scope: it can use compact positive-only storage such as `R11G11B10F` for alpha-free RGB bloom intermediates, but it does not change the main scene target. The counters distinguish `r_bloom 1` pre-final bloom from `r_bloom 2` final-pass bloom, record requested versus effective `r_bloom_passes`, retain the active `r_bloom_blend_base`, `r_bloom_filter_size`, `r_bloom_threshold`, `r_bloom_threshold_mode`, `r_bloom_soft_knee`, `r_bloom_modulate`, `r_bloom_intensity`, and `r_bloom_reflection` values, and expose frame-feature counters for render-scale, greyscale, window-adjusted output, screenshots, and minimized frames. The color audit in [GLX_COLORSPACE_AUDIT.md](GLX_COLORSPACE_AUDIT.md) now defines authored color textures as sRGB only for active scene-linear color, lightmaps and utility textures as linear/data, scene/bloom FBOs as linear numeric targets when scene-linear color is active, screenshots/video as SDR sRGB captures, `r_screenshotCaptureMode` as an explicit HDR-aware policy marker that preserves SDR defaults, and `GL_FRAMEBUFFER_SRGB` as disabled on the current shader-encoded final path to avoid double encoding. GLx also reports per-frame color-pipeline CSV/JSON with `r_glxColorPipelineDebug`, compact texture decode manifests, output-contract state, output primaries/gamut intent, requested versus resolved HDR precision, selected bloom storage, post-shader feature/hash diagnostics, post-shader cache compile/link/source diagnostics, effective LUT atlas size from the shared postprocess executor, screenshot color/luma histograms, final-frame luminance false-color PNG sidecars, generated [visual dossiers](GLX_VISUAL_DOSSIER.md), and versioned `postProofEvidence` for greyscale/render-scale proof in runtime artifacts; diagnostic gates require those color-frame and postprocess rows instead of treating them as optional debug noise. The platform output layer now exposes SDL display HDR state, window HDR headroom, SDR-white hints, ICC profile availability, Windows scRGB/HDR10 capability hints, macOS EDR headroom, and Linux experimental HDR state behind explicit compositor/protocol checks. `r_outputBackend` records the requested backend (`auto`, SDR sRGB, Windows scRGB, HDR10/PQ, macOS EDR, or Linux experimental HDR telemetry/prototype), while `r_outputAllowExperimentalLinuxHDR` keeps Linux HDR opt-in until compositor support is validated. GLx carries the selected/native output backend in `OutputTransform`, `glxpostprocess`, and compact `r_speeds 7` output; Vulkan maps `r_outputBackend 3` or `r_hdrDisplay 1` to its HDR10 swapchain negotiation and reports the same platform-output state in `vkinfo`. Modern GLx tiers record grade and tone-map post nodes separately so the final pass order remains inspectable; when `r_glxPostShaderExecute 1` is active and the shader binds, bloom-prefinal, bloom-final, greyscale, gamma-direct, gamma-blit, screenshot, and hardware-HDR output nodes report executable ownership, with legacy fallback reasons retained for disabled, unbound, or unsupported cases.

On GL3X, GL41, and GL46, the postprocess front end now builds an explicit GLx post/output plan for the frame, emits the plan's `PostNode` chain and `OutputTransform` into the GLx executor for validation and fingerprinting, and records why the plan fell back when the FBO, shader program, output contract, minimized state, disabled experimental executor, failed shader bind, or executor validation rejects ownership. The experimental binder can execute the cached GLSL post shader for display-referred SDR legacy gamma, scene-linear SDR, greyscale, bloom-final, bloom-prefinal, screenshot/blit, Display P3/EDR, scRGB/linear HDR, and HDR10/PQ output when `r_glxPostShaderExecute 1` is set; diagnostics report candidates, eligible frames, binds, fallbacks, and reject masks separately from the broader post/output ownership plan. Older tiers retain the compatibility fallback policy (`GL12` fixed-function output and `GL2X` postprocess-lite`). Modern ownership-proof diagnostics must distinguish planned products from executable products: current builds should show positive planned and executable post-node/output/plan fingerprints when the experimental executor is enabled, while disabled or failed bind paths remain visible as fallback masks until promotion proof accepts the broader surface.

## GLx Profiles

`r_glxProfile` freezes the narrow GLx startup profile used by the release-candidate gates. It is intentionally separate from the lower-level developer cvars so automated sweeps and human testing can select the same profile before renderer registration, world VBO upload, static-world arena setup, and FBO-backed resources are built. The runtime `rc` table and the sweep harness `glx-parity` profile are compared by tests so profile drift fails in automation; `python scripts/glx_runtime_sweep.py --list-profiles` prints the exact cvar contract.

| Profile | Purpose |
|---|---|
| blank / `manual` | Leave individual cvars alone for focused developer experiments. |
| `off` | Restore compatibility defaults for the cvars owned by the GLx profile. |
| `rc` | Conservative release-candidate surface: VBO world cache, GLx world renderer bundle, GLx-owned static arenas, device and soft static draw dispatch, ordered packet-batch static spans, same-state static multidraw, capability-gated high-end indirect buffer/single-draw/MDI span submission, stream draw for proven stage shapes including depth-fragment first passes, CPU-computed texmods, environment coordinates, GL3X+ dynamic-light `auto` streaming with render-IR dlight ownership proof, state-only shadow volumes, immediate beams, and postprocess draw arrays, material renderer, final-pass bloom parity, and GPU timing. Compact per-batch MDI uploads and high-risk screen-map/video-map material gates stay off. |
| `stress` | `rc` plus compact visible-command MDI uploads for non-adjacent full-packet batches. |

The `glxprofile` console command inspects and applies those same profiles: `glxprofile status`, `glxprofile rc`, `glxprofile stress`, `glxprofile off`, and `glxprofile manual`. For startup-stable testing, prefer `+set r_glxProfile rc` or `+set r_glxProfile stress` before the renderer loads. Applying a profile from the console is still useful, but VBO/FBO-backed resources need a map reload or `vid_restart` before every part of the profile can be rebuilt under the new settings.

## Runtime Switching and Sweep Harness

Dynamic renderer builds now expose `renderer_switch <renderer> [fast|keep_window|full|destroy_window]`. It validates the renderer name, force-updates `cl_renderer`, and restarts the video subsystem through the existing `CL_Vid_Restart` path. A changed renderer still causes `CL_ShutdownRef` to unload the active renderer DLL, so switching exercises the same teardown, context/window handling, renderer registration, VM restart, screenshot, and cgame recovery behavior that manual `cl_renderer` plus `vid_restart` testing depends on. A requested `glx` load now fails closed if the GLx renderer DLL is missing, so sweep evidence cannot accidentally come from the legacy renderer.

`scripts/glx_runtime_sweep.py` automates the compatibility smoke pass that GLx needs before promotion. It writes generated configs under an isolated `.tmp/runtime-sweeps/<run-id>/home` fs_homepath, launches a built client with a selected renderer sequence, loads maps, switches renderers in-process, captures named PNG screenshots, runs GLx diagnostics after GLx captures, and records a JSON manifest plus logs. Demo sweeps run each selected renderer in its own timedemo process and rely on `nextdemo "quit"` so completed demos terminate cleanly.

The manifest now records a versioned `rendererSwitchEvidence` object for that
switch run. It expands the selected maps, switch rounds, and switch sequence
into expected map/round/step/renderer tuples, verifies that each tuple produced
a found screenshot, records the `fast` keep-window `CL_Vid_Restart` path used
by the generated config, and ties GLx diagnostics plus `r_speeds 7` performance
samples to the GLx leg. Named non-dry-run RC gates reject missing, stale, or
incomplete lifecycle evidence before treating screenshot or timedemo artifacts
as proof.

Typical local run from the repository root:

```sh
python scripts/glx_runtime_sweep.py --exe meson/build/fnql.x64.exe --basepath meson/build --renderers opengl,glx --switch-sequence opengl,glx,opengl,glx --maps q3dm1 --demos demo1
```

Use `--profile baseline` for a conservative renderer-switch check, `--profile glx-parity` for the current RC candidate surface backed by `r_glxProfile rc`, `--profile glx-ownership` for the same RC surface with `r_glxRequireOwnership 1` and hard failure on any legacy draw delegation, or `--profile glx-stress` to include compact static-world MDI uploads through `r_glxProfile stress`. Unless `--maps` is provided, the ownership and stress profiles capture both `q3dm1` and `q3dm17` so the sweep covers a small stock map and a larger static-geometry scene. `--dry-run` writes the configs and manifest without launching the engine, which is useful on machines without retail assets installed. The manifest records both the full expanded profile cvars and the smaller startup/config cvar sets used to launch the client. `--list-profiles` prints the frozen profile contracts, and `--proof-dir` wires reviewed screenshot/performance baselines plus diff and summary outputs for hard RC evidence. VS Code also has a `Release: Runtime Sweep x64 GLx` task that builds the x64 GLx client and runs the default `q3dm1` screenshot switch sweep.

The same tool also exposes named RC gate presets:

```sh
python scripts/glx_runtime_sweep.py --list-gates
python scripts/glx_runtime_sweep.py --gate rc-smoke --exe meson/build/fnql.x64.exe --basepath meson/build
python scripts/glx_runtime_sweep.py --gate rc-parity --exe meson/build/fnql.x64.exe --basepath meson/build
```

`rc-smoke` proves renderer lifecycle and screenshot capture. `rc-parity` is the conservative blocking RC gate and fails if the renderer-switch lifecycle lacks an `opengl -> glx -> opengl -> glx` round trip, if versioned `worldProofEvidence` does not prove the stock-map/lightmap GLx screenshot and static-world counter floor, or if GLx timedemo FPS drops below 90% of the legacy `opengl` run for the same demo on the same machine. `rc-proof` is the hard evidence gate: it runs the same RC surface but requires reviewed screenshot baselines, a compared performance baseline, the same versioned renderer-switch lifecycle evidence, the broader world proof for high-geometry, fog-heavy, and visibility maps, versioned `materialProofEvidence` for material-stage/tcgen telemetry plus guarded screen-map/video-map stream state, versioned `dynamicProofEvidence` for entity/weapon categories, positive dynamic-light stream and render-IR ownership evidence, shadow stream evidence, required dynamic-scene screenshots or timedemos, versioned `postProofEvidence` for greyscale/render-scale FBO, frame, dimension, color-contract evidence, and post shader direct-final diagnostics, plus the versioned screenshot, demo-playback, HUD, shadow, bloom, cel-shading, greyscale, and render-scale parity suites from [GLX_PROOF_CORPUS.md](GLX_PROOF_CORPUS.md). `glx-ownership` profile runs are promotion evidence rather than RC gates; their manifests write versioned `ownershipProofEvidence` so promotion review can verify zero legacy delegation, GL3X+ tier diagnostics, executable GLx-owned post/output counts, post/output fingerprints, and direct-final shader diagnostics from one artifact field. `rc-stress` exercises compact static-world MDI command uploads plus staged animated-image, screen-map, and video-map material proof, staged particle/poly/mark/beam dynamic proof, and the same postprocess proof before those aggressive paths are considered for default use.

The dedicated `glx-verification` GitHub Actions workflow keeps that gate surface exercised in CI. Hosted jobs build and run `fnql_glx_logic_tests`, then dry-run every RC gate and upload the generated configs, manifests, logs, and Markdown summaries. Runtime jobs run on self-hosted GPU runners with retail assets: manual dispatch can target any gate and proof platform, while the scheduled mainline job runs `rc-parity` weekly from repository variables. Those non-dry-run artifacts are the ones that count toward the blocking RC evidence in [GLX_RC_GATES.md](GLX_RC_GATES.md), and tagged release packaging now revalidates the reviewed proof root before writing release archives.

For visual regression checks, `glx_runtime_sweep.py` can also compare captured PNG screenshots against an approved baseline directory with `--screenshot-baseline-dir`, write difference images with `--screenshot-diff-dir`, and gate the run with `--screenshot-max-rms` plus `--screenshot-max-pixel-ratio`. Use `--approve-screenshot-baselines` only when deliberately refreshing a reviewed baseline set. `--proof-dir` is the preferred RC proof shortcut because it derives the screenshot baseline directory, performance baseline path, diff directory, and Markdown summary path consistently.

Non-dry-run gate manifests also analyze GLx diagnostic output from the screenshot sweep. Material readiness and shader failures, FBO/bloom failures, missing greyscale/render-scale postprocess evidence, minimized output, invalid output color contracts, dynamic-stream failure counters, static-world readiness loss, and static GL errors become explicit gate failures; ordinary compatibility fallback and packet-shape counters remain recorded for review rather than blocking by themselves.

The sweep also enables `r_speeds 7` briefly around each GLx screenshot capture unless `--no-perf-samples` is supplied. Those compact `glx:` frame-counter lines are parsed into manifest and Markdown performance summaries, covering the locked pass-schedule proof line, draw/index pressure, GPU timer text, stream strategy/readiness, material failure counts, postprocess output, stream draw pressure, material-shape and state-only stream draw counts, and static-world counters. Named RC gates require at least one non-dry-run GLx performance sample with the canonical pass schedule.

Those samples are now gateable evidence rather than passive telemetry. Named gates apply a built-in zero-tolerance budget for stream rejects, same-frame stream wrap rejects, material failures, streamed/static draw fallbacks, and static MDI errors, plus tier-selected ceilings for draw calls, index pressure, stream upload volume, material/shader binds and switches, static packet misses, and GPU frame time. `GL3X`, `GL41`, and `GL46` samples must include numeric GPU frame time. `--performance-budget` can add or override runner-specific global or per-tier `max`/`min` thresholds. `--performance-baseline` compares aggregate counter growth against an approved JSON baseline, while `--approve-performance-baseline` refreshes that baseline only during an intentional review run. `rc-proof` rejects approval mode and requires actual comparisons, so manifests distinguish reviewed proof from baseline-refresh runs.

`fnql_glx_logic_tests` provides the first renderer-focused pure logic test target. It runs without an OpenGL context, without the legacy `code/renderer` include path, and without retail assets. It covers the RC material permutation key classifier, `MaterialIR` to material state-plan compilation, explicit unsupported material compiler reasons, depth-fragment as a material feature for single-texture and multitexture first passes, special scene flags staying gate-only instead of expanding the shader permutation table, unsupported multitexture combine rejection, five-tier product-tier/version/extension logic, stream strategy fallback selection, stream draw material-gate allowlists including explicit dynamic-light, screen-map, video-map, shadow-volume, and beam gating, static-world packet classification, static draw-policy gating, and the developer-only broad key-mode escape hatch. `fnql_glx_header_boundary` complements it by scanning the pure GLx headers, the renderer-common GLx API/forwarding bridge, and the renderer facade for accidental dependencies on `qgl`, `tr_public.h`, renderer ABI implementation types, OpenGL object typedefs, module-local headers, or direct `code/rendererglx` includes from `tr_glx_compat.h`:

```sh
meson compile -C meson/build fnql_glx_logic_tests
meson test -C meson/build fnql_glx_logic fnql_glx_header_boundary --print-errorlogs
```

`r_glxStreamDrawKeyMode` controls the stream material gate used after the hard eligibility checks and `MaterialIR` compiler validation:

- `0`: plain keys plus any explicitly enabled special-case gates, still rejecting ungated texmods, environment coordinates, and special dynamic bundles.
- `1`: computed keys, allowing texmods as a compatibility shorthand while still rejecting video, screen-map, dynamic-light, and environment cases unless their explicit gates are enabled.
- `2`: all currently eligible single-texture generic stages.

For RC testing, prefer `r_glxStreamDrawKeyMode 0` plus the explicit material gates (`r_glxStreamDrawMultitexture`, `r_glxStreamDrawFog`, `r_glxStreamDrawDepthFragment`, `r_glxStreamDrawTexMods`, and `r_glxStreamDrawEnvironment`) so each compatibility shape can be measured and backed out independently. Dynamic-light uses its own `auto` gate on GL3X+ and must prove matching render-IR dlight role/pass ownership; shadow-volume, beam, and postprocess draw-array gates are part of the RC profile because they submit already-prepared state-only payloads. Screen-map and video-map material gates remain off. `r_glxStreamDrawKeyMode 2` remains a developer escape hatch for broad experimentation, not an RC profile.

Stream draw counters report accepted/rejected material keys, material-compiler rejection counts with the last unsupported reason, and skip reasons for ineligible stages, including missing buffer binding support, multitexture, depth-fragment gating, missing texcoords, empty batches, material-key rejection, and fog-gate rejection. They also split out second-texture-coordinate upload pressure, multitexture and depth-fragment gate accepted/rejected counts, multitexture draw count, fog draw count, depth-fragment streamed stage count, texmod accepted/rejected/drawn counts, environment accepted/rejected/drawn counts, dynamic-light auto-gate counters, the still-disabled screen-map/video-map gate counters, and the RC-profile shadow/beam/postprocess draw-array counters. Per-feature accepted/rejected counters describe whether that present feature passed its own gate; a material key can still be rejected by another present feature. The generic stage path records the material gate decision before returning through the hard multitexture or depth-fragment skip reasons, so those compatibility-sensitive rejections remain visible in both gate counters and skip counters.

Draw telemetry is recorded from the existing renderer hot paths: client-array draws through `R_DrawElements`, static-world device-index VBO draws, and static-world soft-index VBO draws. Material-stage telemetry is recorded from the generic and VBO stage iterators before each legacy pass binds state and submits indexes. It tracks fixed-function state bits, color/alpha generators, texture coordinate generators, texmod counts, multitexture, depth-fragment, blend, alpha-test, depth-write, lightmap, animated-image, video, screen-map, dynamic-light, and environment flags, plus a small fixed hot-key table by index pressure. `r_glxDebug 1` asks SDL and WGL startup paths for a debug OpenGL context when `cl_renderer glx` is active, falling back to a regular context if the platform cannot provide one. `r_glxDebugGroups 1` wraps shader batches in `KHR_debug` groups when the extension functions are available, while `ARB_debug_output`-only drivers still get callback logging without object labels or groups.

Shader-batch telemetry is recorded at `RB_EndSurface` before the existing stage iterator runs. It tracks batch count, vertex/index pressure, total stage passes, pass-count histogram, sort histogram, fog/multitexture/polygon-offset flags, VBO vs generic batches, largest batch, and a small fixed top-shader table by index pressure. Material-stage telemetry complements that batch view with per-pass keys so later streamed draw and GLSL permutation work can target the most common compatibility shapes first.

Static-world packet telemetry is recorded when the legacy VBO builder has packed static BSP surfaces by shader at map load. GLx now snapshots one packet per sorted shader group with shader name, sort, surface/vertex/index pressure, packed vertex/index offsets and byte spans, stage-pass count, and batch flags. It also derives one `DrawElementsIndirect`-shaped command record per valid packet, including index count and first-index offset, while reporting invalid or misaligned packet records separately. Packet output includes the compact command index so packet-owned submission can move from a manifest packet to a command-buffer slot deterministically. `r_glxStaticWorldIndirectBuffer 1` uploads the compact valid-command list into a `GL_DRAW_INDIRECT_BUFFER` when the driver exposes draw-indirect support, preserving the previous binding and labeling the buffer for GL debuggers.

`r_glxWorldRenderer 1` enables the parity-oriented static-world GLx bundle. It implies static arena upload, arena binding, device-index run dispatch, ordered multidraw batching when available, soft-index dispatch, and an effective `all` draw policy for device runs. The compatibility renderer still owns world visibility, shader sorting, stage iteration, fog setup, texture binding, ARB program setup, debug overlays, and fallback order; GLx owns the buffer/draw submission edge and routes unsupported draws through the GLx legacy-delegation draw bridge. This is the intended switch for whole-world parity testing because it avoids needing to hand-compose the lower-level static-world cvars. It requires the static VBO world cache, so use it with `r_vbo 1` before map load or `vid_restart`.

`r_glxStaticWorldArena 1` additionally uploads the already-packed legacy static world vertex and index payloads into GLx-owned buffer objects after the legacy VBO upload succeeds. This does not draw anything by itself and it does not replace `VBO_world_data` / `VBO_world_indexes`; it is the shipped static-world ownership bridge so GLx can validate buffer lifetime, memory pressure, and binding preservation before every static-world draw takeover. It is enabled explicitly by the RC profile and is also implied by `r_glxWorldRenderer`.

`r_glxStaticWorldArenaDraw 1` lets the existing legacy VBO stage iterator bind the GLx-owned static arena buffers when they are available. Shader state, sort order, queue preparation, and draw calls still use the compatibility renderer path; only the bound vertex/index buffer object handles change. If the arena is not ready, GLx reports a draw skip and the bind helpers keep using the legacy buffers. It is implied by `r_glxWorldRenderer`.

Static-world queue telemetry now classifies every prepared device-index run against the packet manifest before any static-world draw cvar is considered. GLx builds a fixed item-index-to-packet lookup while recording the manifest, uses it for the common first/last item range classification path, and falls back to byte-span scanning when a range is missing, overflowing, or inconsistent. It also measures how visible full-packet runs line up with the compact indirect command buffer: contiguous command runs, command count, covered indexes, breaks, and largest adjacent command run are reported for both the last queue and cumulative totals. `glxstaticworld` and `r_speeds 7` report full-packet, partial-packet, miss, item-mismatch, lookup-hit, lookup-fallback, and indirect-command adjacency counts so the next packet-level draw work can be guided by real visibility behavior instead of assumptions. The packet list also accumulates per-packet queue and draw heat (`q` full/partial runs, `d` full/partial draws, and manifest-owned draw indexes), while `glxstaticworld hot [limit]` ranks packets by weighted queue/draw pressure to identify stable candidates for later packet-level submission.

`r_glxStaticWorldDraw 1` routes static-world device-index VBO runs through the GLx static draw dispatcher after the legacy renderer has prepared queue runs and bound the stage state. The dispatcher validates each submitted byte range and queued item range against the GLx static packet manifest, records full-packet, partial-packet, miss, item-mismatch, and arena-vs-legacy-buffer counters, issues the same `glDrawElements` call, and returns control to the VBO iterator for later shader stages. If disabled or rejected, the remaining VBO draw is submitted through the GLx legacy-delegation draw bridge and counted by the ownership diagnostic. It is implied by `r_glxWorldRenderer`.

`r_glxStaticWorldSoftDraw 1` routes the short-run CPU-index fallback path through GLx as well. These draws intentionally do not use packet substitution because the software index buffer can contain a stitched list of visible items; GLx submits the same client index span that the legacy VBO path would have submitted and reports soft-draw call/index/fallback counters. It is implied by `r_glxWorldRenderer`.

`r_glxStaticWorldDrawPolicy` defaults to `full`, which means GLx only submits runs that exactly match one manifest packet. For full-packet runs, GLx issues the draw from its own manifest packet offset and index count rather than the legacy run fields, so the static-world handoff is now packet-owned while still protected by legacy queue validation. `contained` also allows partial visibility ranges that remain inside a packet, while `all` submits any valid legacy device run and is mainly useful for parity and broad dispatcher comparisons. `r_glxWorldRenderer` uses an effective `all` policy. Policy skips are counted separately from GL function or validation fallbacks.

`r_glxStaticWorldIndirectDraw 1` lets the single-run GLx dispatcher submit full manifest-packet static draws with `glDrawElementsIndirect` when `r_glxStaticWorldIndirectBuffer` has successfully uploaded the command buffer and the capability table advertises draw-indirect support. It is stricter than the normal GLx static draw dispatcher: partial packet runs, item mismatches, missing command slots, missing function pointers, or GL errors fall back to the existing manifest-owned `glDrawElements` path. The draw-indirect binding is restored around each submission.

`r_glxStaticWorldMultiDraw 1` lets GLx collapse eligible same-state static device-index runs from one VBO queue into one or more `glMultiDrawElements` calls when the driver exposes it. Each run carries its original VBO item range so GLx can classify whether visibility produced complete shader packets or partial packet runs. Multidraw is now filtered instead of all-or-nothing: GLx marks only the runs it actually submits, and the VBO iterator keeps rendering the remaining runs through the single-run GLx dispatcher or the GLx legacy-delegation per-run draw path. Non-eligible runs are treated as order barriers: GLx flushes any pending eligible batch before the barrier and leaves that barrier plus later runs to the compatibility queue, so the filtered path cannot move a later fast-path draw ahead of an earlier fallback draw. With broader policies such as `contained` or `all`, partial-but-allowed runs can still stay inside the ordered GLx span walker. It is implied by `r_glxWorldRenderer`.

`r_glxStaticWorldPacketBatch 1` adds the conservative packet-level batch step used by the RC profile. After the legacy VBO queue has already sorted visible static surfaces and the shader stage has bound its normal compatibility state, GLx splits each eligible filtered batch into ordered spans. Full manifest-packet spans submit from the packet manifest's index counts and offsets with `glMultiDrawElements`; partial, contained, or legacy-shaped spans keep their original run offsets through the existing GLx fallback draw path. One-run spans are submitted immediately in queue order, so the packet path cannot move a later full packet ahead of an earlier fallback run. The result is a packet-owned submission edge for common full-packet visibility in the shipped profile.

`r_glxStaticWorldMultiDrawIndirect 1` adds an even narrower filtered-batch path on top of `r_glxStaticWorldMultiDraw`: if every run in a flushed filtered batch is a full manifest packet and those packets map to adjacent command-buffer slots, GLx submits that batch with one `glMultiDrawElementsIndirect` call. If the batch is partial, missing a command, non-adjacent, unsupported, or hits a GL error, GLx falls back to the existing `glMultiDrawElements` filtered batch without changing the draw order.

`r_glxStaticWorldMultiDrawIndirectCompact 1` lets that same MDI gate upload a compact per-batch command list for full manifest packets that are visible in a non-adjacent order, or when the static command buffer was not uploaded. This preserves the VBO queue order by copying only the selected packet commands into a small `GL_DRAW_INDIRECT_BUFFER` immediately before the call. The compact command buffer is reused when possible: after an initial `glBufferData` allocation, later batches use `glBufferSubData` if the driver exposes it and the existing scratch capacity is large enough. When KHR_debug object labels are available, the lazily-created compact command buffer is labeled after the filtered submission path observes it. It remains separately gated because it trades fewer draw calls for per-batch command-buffer uploads and needs real-map telemetry before becoming a preferred path.

`r_glxStaticWorldMultiDrawIndirectSpans 1` lets GLx split one filtered static multidraw batch into ordered spans before submission. Consecutive full-manifest spans try the MDI path, while consecutive non-manifest spans use the existing `glMultiDrawElements` fallback; if no useful span is found, GLx keeps the original whole-batch filtered multidraw behavior. This is useful with broader draw policies such as `contained` or `all`, where one partial packet should not prevent neighboring full-packet spans from exercising the indirect path. One-run outer spans are now submitted immediately in queue order instead of being deferred to the caller's later per-run fallback loop. When compact MDI is disabled but the static indirect command buffer is ready, full-manifest spans can also split into adjacent command subspans. Singleton command gaps are submitted in order through the existing `r_glxStaticWorldIndirectDraw` single-draw path when it is enabled and ready, otherwise as ordinary GLx single draws, then neighboring adjacent command spans can still use MDI without perturbing the original visible order. The MDI submitter now refuses the path unless the one-time GLx capability table advertises multi-draw-indirect support, rather than trusting function-pointer presence alone.

The MDI counters intentionally split rejection reasons into unsupported/missing command-buffer state, short batches, non-manifest packet runs, missing command slots, non-contiguous command spans, compact upload failures, and GL errors. They also remember the most recent rejection reason, rejected run/index pressure, rejected command slot, last outer span shape, largest outer span split, how many one-run spans were submitted directly or through single-run indirect draws, and the concrete command ranges from the most recent visible queue. `glxstaticworld spans [limit]` prints this focused view so real-map tuning can distinguish capability gaps, visibility-order gaps, compact-upload pressure, and actual GL failures without stepping through the renderer.

The MDI diagnostics also remember the last successful source (`static` command buffer or compact upload), last run/index pressure, first static command slot when applicable, largest index span, and the last command-subspan split shape. This makes runtime tuning possible from `glxstaticworld` or `r_speeds 7` without stepping through the renderer.

Static-world queue telemetry is recorded after the legacy VBO queue sorter has grouped visible static surfaces for a shader batch. It tracks total prepared queues, last queued item/index pressure, device-index run count, soft-index fallback pressure, and largest contiguous device-index run. This measures the exact legacy split that a future GLx-owned static arena or indirect draw path must preserve before replacing it.
