## Implementation gaps and improvement opportunities

The largest gap is **zone authoring sophistication**. The audio-zone compiler is currently built around maintainer-authored text files and axis-aligned boxes with per-zone preset/gain/bias values, and the runtime chooses zones by containment, priority, and smaller-volume tie-breaks. That is pragmatic and safe, but it is fundamentally a manual AABB system, not a geometry-derived acoustic partitioning system. It does not yet infer rooms from BSP leaves, model portals or door-connected spaces, classify surface absorption, or generate zones automatically from existing maps. Against the userÔÇÖs desired end-state, this is the clearest implementation shortfall. ţłÇfileciteţłéturn18file0ţłü ţłÇfileciteţłéturn16file0ţłéL1-L1ţłü ţłÇfileciteţłéturn19file0ţłü

Related to that, the fallback environmental model is still heuristic. It uses point contents, a small set of ray probes, sky-hit detection, horizontal openness, and hallway/outdoor/underwater classification to choose reverb families. That is a respectable default and a good safety net, but it is not a substitute for room graph analysis or material-aware acoustic tagging. The user specifically asked for ÔÇťrobust and high-quality zone mapÔÇŁ compilation for existing maps; the current heuristic path is useful, but it is not robust enough to be the final answer for that requirement. ţłÇfileciteţłéturn19file0ţłü

Surround support is **good but not yet exhaustive**. The backend now requests output modes including headphones, speakers, quad, 5.1, 6.1, and 7.1; supports direct-channel playback for non-spatial stereo/multichannel material; and supports multichannel PCM paths when the device exposes `AL_EXT_MCFORMATS`. That is already much better than a stereo-only implementation. However, the present engine surface still stops short of a truly comprehensive surround productization story: there is no engine-level speaker-layout validation suite for every output mode, no first-class exposure of advanced surround decode/encoding options, and no higher-order content path beyond what OpenAL Soft can do implicitly. ## Roadmap to make the system best-in-class

The first priority should be to move from ÔÇťplan completedÔÇŁ to ÔÇťquality ceiling raised.ÔÇŁ That means preserving the current architecture, not replacing it. The present OpenAL-based design is the right one for FnQL because it already delivers modern positional audio, HRTF, environmental FX, surround-capable output modes, and safe backward compatibility without requiring new assets or a new runtime dependency stack. OpenAL Soft remains a strong fit for this requirement set, and the current implementation is already aligned with that choice. ţłÇfileciteţłéturn3file0ţłéL1-L1ţłü ţłÇciteţłéturn3search2ţłéturn6search0ţłéturn12view0ţłéturn12view1ţłü

The next concrete phase should focus on **surround and playback polish**. The engine should add a dedicated speaker-layout validation suite that renders deterministic test assets for stereo, quad, 5.1, 6.1, and 7.1 and verifies channel routing, direct-channel behavior, and HRTF/non-HRTF mode changes through loopback or capture harnesses. The same phase should modernize the stream path so that raw/music playback is either queued at the actual mixer frequency or resampled using a better-quality SRC. OpenAL SoftÔÇÖs own documented resampler and buffering controls are a strong external reference for what ÔÇťhigh quality but configurableÔÇŁ looks like here. ţłÇfileciteţłéturn20file0ţłü ţłÇfileciteţłéturn21file0ţłü ţłÇciteţłéturn8view0ţłéturn8view2ţłéturn10view0ţłü

The phase after that should build the **real zone-system answer** that the user asked for. The target should be an **automatic zone compiler for existing BSP maps** that begins with geometry extraction, BSP leaf/cluster parsing, portal/door adjacency analysis, and content/surface classification, then generates candidate acoustic volumes automatically. Those candidates should then be refined through rule-based merging and optional maintainer overrides before being compiled into a richer binary format. ## Recommended priority order

The highest-value order is clear.

First, keep the existing architecture and close the obvious audio-quality gaps: better stream SRC, stronger automated surround validation, and more explicit regression tests around direct-vs-spatial routing and multichannel behavior. Second, replace the current manual zone workflow with a BSP/material-aware compiler that can generate high-quality defaults for existing maps and only then accept optional human overrides. Third, improve runtime polish through safer live reconfiguration, better device-loss behavior, and better platform guidance for OpenAL Soft configuration. Only after those steps should the project consider more ambitious optional work such as richer ambisonic/surround authoring paths or deeper acoustic simulation techniques. ţłÇfileciteţłéturn20file0ţłü ţłÇfileciteţłéturn21file0ţłü ţłÇfileciteţłéturn16file0ţłéL1-L1ţłü ţłÇfileciteţłéturn18file0ţłü ţłÇfileciteţłéturn19file0ţłü ţłÇciteţłéturn3search2ţłéturn10view0ţłéturn13view0ţłü

## Implementation progress

As of May 8 2026, the first pass through that priority order has been implemented in the branch. Stream playback now targets the active mixer rate with cubic SRC, loopback coverage exercises HRTF switching plus stereo/quad/5.1/6.1/7.1 direct-channel paths, the OpenAL device layer reports and can recover from supported disconnect/reopen events, and opt-in UHJ/B-Format sample tags are accepted when the runtime exposes the matching OpenAL extensions.

The zone pipeline has also moved beyond the manual AABB baseline described earlier in this review. `fnql-audiozonesc --from-bsp` now derives generated zones from existing BSP data, classifies material and outdoor/underwater hints, writes version 2 sidecars, and preserves manual override priority. Runtime loading still accepts version 1 sidecars, but version 2 metadata is now consumed: material and flag metadata are retained for diagnostics, outdoor/underwater flags feed the existing tone-class behavior, and generated portal hints provide a bounded blend toward adjacent zone environments near shared boundaries. Missing or invalid sidecars still fall back to generic heuristics.

The remaining high-value follow-up is validation depth rather than a missing primary feature: focused `.azb` parser/runtime unit coverage now exists, `fnql-audiozonesc --audit` provides a deterministic preflight for generated sidecar coverage, warnings, and lookup/portal-blend profiling, and `fnql_audio_recovery` covers the device-loss policy without requiring physical hardware disconnects. The next validation work is listening sweeps on real maps plus manual device-loss checks across Windows, Linux, and macOS hardware.

## Open questions and limitations

This review is based on direct repository inspection and external technical references, not on listening tests or runtime profiling on target hardware. The biggest implementation conclusions are high-confidence, but the exact severity of the stream-fidelity issue, hotplug/device-loss ergonomics, and platform-specific runtime behavior should still be validated with real playback and loopback captures on Windows, Linux, and macOS.

The zone pipeline is no longer the clearest missing feature. The next meaningful investment is proving and tuning it with `--audit` outputs, real-map listening passes, and manual platform sweeps for modern OpenAL device recovery.
