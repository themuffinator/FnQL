# GLx Renderer Implementation Plan

## Mission and scope

FnQL explicitly positions itself as a modernized Quake III Arena source port that should improve the experience without changing the game code, and it ranks its priorities in this order: retail/demo compatibility, speed and efficiency, modern platform support, and cross-platform reach. That priority stack strongly favors a new renderer module that preserves the existing engine boundary and modernizes implementation behind it, rather than a broad engine rewrite. The repository already supports modular renderer selection through `cl_renderer`. In practice, that means GLx should be treated as the consolidating replacement for the OpenGL lineage, not as a fourth forever-renderer that adds even more maintenance. ## What FnQL already tells us

The renderer ABI is already in the right shape for this project. `REF_API_VERSION 8`, the `GetRefAPI` entry point, the `refimport_t` / `refexport_t` split, engine allocators, and platform callbacks already isolate renderer internals from the rest of the engine. That means GLx can be internally modern C++ while remaining a C ABI module at the boundary, which is exactly the architecture you want for a compatibility-first port. ŅłĆfileciteŅłéturn29file0ŅłéL1-L1Ņłü

The repo guardrails are also clear. `AGENTS.md` and the technical notes both treat demo behavior, protocol behavior, asset loading, VM ABI, and renderer defaults that affect demo output or determinism as compatibility-sensitive. They also favor incremental engine changes and require performance regressions to be justified and measured. The implication is important: GLx can be a major local rewrite inside one renderer module, but integration with the rest of FnQL should stay conservative and reviewable. ŅłĆfileciteŅłéturn32file0ŅłéL1-L1Ņłü ŅłĆfileciteŅłéturn33file0ŅłéL1-L1Ņłü

The current OpenGL lineage shows why consolidation is worth doing. The legacy renderer still does manual extension discovery and symbol resolution, uses ARB program-era machinery, and only conditionally enables later features such as VBO/FBO support. Renderer2 adds much more modern machinery, but it also exposes a huge feature surface with toggles for postprocess, HDR, float lightmaps, SSAO, cubemaps, parallax, shadowing, and PBR-related behavior. Maintaining multiple OpenGL implementations with that degree of divergence is exactly the kind of redundancy GLx should remove. ŅłĆfileciteŅłéturn22file0ŅłéL1-L1Ņłü ŅłĆfileciteŅłéturn23file0ŅłéL1-L1Ņłü

There are also concrete implementation lessons already present in FnQL. Renderer2 packs attributes into VAOs/VBOs, maintains a dynamic tessellation VAO, and updates it with buffer orphaning plus `BufferSubData`, which improves submission but still preserves the classic CPU-built `tess` path. The Vulkan backend goes further for static world data: it explicitly classifies static surfaces, stores them in device-local memory, queues items instead of relying on runtime tessellation for those surfaces, and notes that this removes runtime tessellation limits. That is the key design direction for GLx: keep id Tech 3 compatibility, but move static scene submission much closer to the Vulkan strategy. ŅłĆfileciteŅłéturn17file0ŅłéL1-L1Ņłü ŅłĆfileciteŅłéturn26file0ŅłéL1-L1Ņłü

FnQL already has a front-end/back-end render command list and existing performance counters for shaders, batches, surfaces, VAO draws, and GLSL binds. GLx should extend those counters rather than inventing a second telemetry layer. The project already has the notion of measured hot paths; GLx should plug into that culture from day one. ## Recommended GLx architecture

Structurally, GLx should live in a new `code/rendererglx/` module and export the same renderer ABI through an `extern "C"` `GetRefAPI` shim. Internally, it should be implemented as modern C++ with one owning `RendererModule` object and clear subsystem boundaries for platform/context management, capabilities, state tracking, shader/material compilation, static world caching, dynamic streaming, pass scheduling, and profiling. Because the repo still documents Visual Studio 2017+ and broad GNU/Clang make-based builds across Windows, Linux/BSD, macOS, Raspberry Pi OS, and PowerPC, the safest modernization baseline is C++17 rather than a 4.5-era OpenGL dependency or an aggressively high C++ language baseline. ŅłĆfileciteŅłéturn29file0ŅłéL1-L1Ņłü ŅłĆfileciteŅłéturn34file0ŅłéL1-L1Ņłü

At the coding level, the C boundary should stay minimal and boring. Everything exported to the engine remains plain-old C structs and function pointers. Internally, use RAII wrappers for GL objects, strongly typed enums for state and pass keys, allocator adapters that route engine-owned lifetime through `ri.Hunk_*` and `ri.Malloc`, and explicit error paths that terminate through `ri.Error` instead of allowing exceptions to cross the ABI. That gets you modern C++ ergonomics without creating hidden ownership problems across the renderer boundary. ŅłĆfileciteŅłéturn29file0ŅłéL1-L1Ņłü

There is also a naming issue worth addressing up front. Khronos already uses **GLX** for the X Window System binding, and the OpenGL registry lists GLX specifications alongside the core API. So the project-internal renderer name can still be ŌĆ£GLxŌĆØ if you want the branding, but the implementation should use unambiguous module and file names such as `rendererglx`, `platform_x11_glx`, `platform_win32_wgl`, and `platform_macos_nsgl` so the code never confuses ŌĆ£the GLx rendererŌĆØ with the actual GLX platform API. ŅłĆciteŅłéturn2search2Ņłü

Architecturally, do **not** start by building a grand general-purpose frame graph. FnQL already has a classic render command pipeline, and compatibility-sensitive renderer defaults mean render order is not a casual detail. The better abstraction is a deterministic pass scheduler that mirrors id Tech 3 expectations: optional depth preparation, opaque world, entities, decals/marks, transparent layers, first-person weapon, 2D/HUD, then postprocess. A lightweight dependency graph is fine inside the postprocess chain, but the whole renderer does not need a modern engine-style frame graph to succeed. ŅłĆfileciteŅłéturn28file0ŅłéL1-L1Ņłü ŅłĆfileciteŅłéturn33file0ŅłéL1-L1Ņłü

The subsystem layout I recommend is straightforward:

- **`Caps`**: version, extension, and quirk detection frozen at init.
- **`PlatformContext`**: context creation, proc loading, swap, debug-context setup.
- **`StateCache`**: blend, depth, raster, texture/sampler, framebuffer, and program state suppression.
- **`ShaderLibrary`**: GLSL source generation, permutation keys, binary cache if later warranted.
- **`StaticWorldCache`**: BSP/lightmap/static-surface packing, batch metadata, and draw packet generation.
- **`DynamicStreamRing`**: per-frame uploads for entities, polys, particles, marks, UI quads, and other transient data.
- **`PassScheduler`**: deterministic pass order with quality-tier feature gating.
- **`Profiler`**: CPU/GPU timings, debug labels, stall counters, upload volume, and regression hooks.

The guardrails should be equally explicit. Do not resurrect fixed-function. Do not make GL 4.5 mandatory. Do not start by porting every renderer2 feature toggle. Do not begin with compute-driven culling, bindless-only submission, or a PBR rewrite. The first goal is a single replacement renderer that is compatible, measurable, and maintainable. Renderer2ŌĆÖs cvar sprawl is precisely why scoping discipline matters. ŅłĆfileciteŅłéturn23file0ŅłéL1-L1Ņłü ŅłĆfileciteŅłéturn32file0ŅłéL1-L1Ņłü

## Capability tiers and performance model

The hard compatibility floor should be **shader-based OpenGL 2.1 with GLSL 1.20**, not fixed-function fallback. Khronos still documents OpenGL 2.1 together with GLSL 1.20, and once you are dealing with modern core-profile style contexts, the fixed-function path is explicitly not the migration direction anymore. If GLx tries to support pre-shader hardware as a first-class target, it recreates the same long-term renderer split it is supposed to eliminate. ŅłĆciteŅłéturn2search2Ņłéturn12search7Ņłü

The preferred default path should be a **core-style GL 3.3+ / 4.1-friendly design**. That gives you a modern enough submission and shader model to simplify the renderer substantially, while still fitting FnQLŌĆÖs platform matrix. This matters especially because AppleŌĆÖs documented OpenGL profiles top out at 4.1 core, so any 4.5-class conveniences have to be opportunistic accelerators, not architectural requirements. ŅłĆciteŅłéturn2search2Ņłéturn12search0Ņłü

Above that, GLx should enable an **advanced acceleration tier** whenever the context exposes the right features or extensions. Khronos provides the exact ladder you need. `ARB_map_buffer_range` adds unsynchronized subrange mapping and explicit flushing for older hardware; uniform buffer objects were added specifically to reduce the runtime overhead of large numbers of `glUniform*` calls and provide stable layouts such as `std140`; instanced arrays directly target repeated-object rendering while reducing API calls and duplicate data; `ARB_buffer_storage` adds immutable storage and persistent/coherent mappings; `ARB_sync` adds fence-style synchronization without forcing a full `glFinish`; `ARB_multi_draw_indirect` allows multiple draws to be dispatched from one call; `ARB_direct_state_access` reduces bind-to-edit traffic; `KHR_debug` provides driver messages, object labels, and debug groups; and timer queries provide GPU timings without stalling the pipeline the way `glFinish()` does. ŅłĆciteŅłéturn15view0Ņłéturn13view0Ņłéturn14view0Ņłéturn4view0Ņłéturn5view0Ņłéturn5view1Ņłéturn5view3Ņłéturn7view0Ņłéturn8view0Ņłü

That leads to a clean performance policy. On every machine, GLx should prepack genuinely static world data at map load and keep it resident in large GPU buffers. On advanced contexts, it should additionally build indirect draw packets and persistent-mapped transient rings. On older contexts, it should fall back first to `MapBufferRange`-style streaming and only then to orphan-plus-subdata behavior. KhronosŌĆÖ own streaming guidance emphasizes that implicit synchronization is the core hazard in dynamic buffer submission, and its vertex best-practice guidance stresses reducing state churn and keeping more meshes in fewer buffers where practical. Combined with FnQLŌĆÖs existing Vulkan static-surface strategy and renderer2ŌĆÖs dynamic tess upload path, the design direction is very clear: static world goes into long-lived caches; transient scene data goes through a ring; state changes and tiny allocations are treated as performance bugs. ŅłĆfileciteŅłéturn26file0ŅłéL1-L1Ņłü ŅłĆfileciteŅłéturn17file0ŅłéL1-L1Ņłü ŅłĆciteŅłéturn9search0Ņłéturn9search3Ņłü

The practical tier model should therefore be:

- **Compat tier**: GL 2.1 / GLSL 1.20, shader-only baseline, conservative postprocess, dynamic streaming via `MapBufferRange` or orphaning, no required persistent mapping or indirect draw submission.
- **Core tier**: GL 3.3+ or equivalent core-style capability set, the primary development target, static world caches always enabled, cleaner state model, stronger shader/material path.
- **Advanced tier**: optional use of immutable persistent buffers, fences, multidraw/indirect submission, debug groups, timer queries, and richer batching paths when the driver exposes them.

The key implementation rule is that this decision happens **once at initialization**. Build a `Caps` table, select function tables and strategy callbacks, and then keep hot paths branch-light. Capability checks sprinkled throughout every draw path are a maintenance trap.

FnQLŌĆÖs stated preference for compatibility-preserving, incremental integration means GLx should land in slices, even if the renderer internals are substantially redesigned. The supporting conditions already exist: modular renderers, a stable ABI, and explicit repo guardrails around compatibility and measured performance. ## Subagent workload and completion criteria

Because the renderer ABI is fixed and the project treats compatibility-sensitive behavior conservatively, agent parallelism should be used aggressively inside the renderer implementation but not for ownership of ABI or render-order decisions. One agent should always own the integration map and review bar. ŅłĆfileciteŅłéturn29file0ŅłéL1-L1Ņłü ŅłĆfileciteŅłéturn32file0ŅłéL1-L1Ņłü ŅłĆfileciteŅłéturn33file0ŅłéL1-L1Ņłü

The workload split I recommend for GPT-5.5 subagents is this:

- **Lead architecture agent**  
  Own the ADRs, capability-tier rules, merge order, milestone board, and ŌĆ£what is allowed to changeŌĆØ document. This agent approves render-order decisions, fallback policy, and default cvar behavior.

- **ABI and build agent**  
  Create the new renderer module, wire build files and project files, keep `GetRefAPI` stable, add `cl_renderer glx`, manage platform-specific compilation, and preserve the modular renderer flow during transition.

- **Context and capability agent**  
  Implement proc loading, context creation, version/extension probing, quirk tables, debug-context creation, DSA/object-label enablement, and the one-time capability function table.

- **Static world agent**  
  Own BSP/lightmap ingestion, static surface classification, GPU arena packing, draw-packet generation, and any indirect-draw command buffers for supported tiers.

- **Dynamic scene agent**  
  Own per-frame geometry upload, transient ring buffers, models, particles, marks, polys, first-person weapon, UI quads, and fallback ladders between persistent mapping, mapped subranges, and orphaning.

- **Shader and material agent**  
  Build the shader-stage translator, permutation key system, uniform/UBO layouts, material caching, compile diagnostics, and shader hot-path profiling hooks.

- **Postprocess and visual parity agent**  
  Own FBO pipelines, internal scaling, HDR precision, gamma, greyscale, bloom parity, cel shading, and all user-facing visual controls that are already documented.

- **QA, performance, and docs agent**  
  Extend `r_speeds`, add GPU timer query instrumentation, create screenshot/diff and demo replay sweeps, track regressions, update `README.md` / display docs / technical notes, and keep migration notes current.

In terms of dependency order, the lead architecture agent, ABI/build agent, and context/capability agent can start immediately. The static world and dynamic scene agents should begin as soon as the capability table and buffer abstractions are frozen. The shader/material agent can start once the pass scheduler and material key format are agreed. The postprocess agent should wait for the base framebuffer and world path. The QA/perf/docs agent should run continuously from the first textured quad milestone onward.

GLx is ready to become the default renderer only when four bars are met. It must load through the existing modular renderer path and survive `vid_restart` / renderer switching; it must preserve the documented display surface, especially the framebuffer controls and the OpenGL-specific bloom behavior unless there is a deliberate, documented replacement; it must pass compatibility-focused screenshot and demo sweeps with stable defaults; and it must show its gains with GPU timer queries plus the engineŌĆÖs own counters rather than intuition. When those conditions are satisfied, `opengl` should become an alias of `glx`, `opengl2` should move behind a legacy build flag, and the maintenance cost of parallel OpenGL renderers can finally be retired.
